# Root Architecture

## Core Synchronization Protocol

1. Update module header docstrings first when responsibilities or dependency direction change.
2. Update the nearest `.package.md` immediately after module-level changes.
3. Update this root map only when top-level layout or architectural boundaries change.

## Top-Level Architecture

- `/src/sentrymode/__main__.py`: CLI entrypoint that wires settings, factor registry, runner, and notifier.
- `/src/sentrymode/monitoring`: shared runtime kernel (settings, protocols, result models, runner, notifier adapter exports).
- `/src/sentrymode/factors`: pluggable signal modules (`ahr999`, `vix`) and registry helpers.
- `/src/sentrymode/market_data.py`: reusable daily-series data access seam used by factor logic.
- `/tests`: CLI-focused regression tests and test fixtures.
- `/scripts`: maintenance utilities for docs and source hygiene.
- `/docs`: strategy and data-source notes; not imported by runtime packages.

## Architectural Constraints

- Factor implementations should depend on `monitoring` protocols/results and optional shared data adapters, not on CLI parsing.
- The `monitoring` package must stay factor-agnostic and avoid importing specific factor modules.
- Configuration ownership stays in `monitoring.settings.Settings`; runtime modules read settings through typed fields rather than ad-hoc env access.
